## 三、关键架构决策

1. **`EnvBackend` 三态抽象**
   `WindowsEnv` / `UnixEnv` / `DryRunEnv`。dryRun 模式只收集 `EnvOp`，`preview()` 出计划，便于单测与审计。

2. **Unix 环境变量托管块模型**
   早期 `UnixEnv` 每次都 append `export PATH`、unset 只写注释，导致**只增不减、累积污染**。改为 rc 托管块（`# >>> devenv-manager >>> … # <<< devenv-manager <<<`）整体重写模型，根治老毛病。

3. **下载进度写盘节流**
   原每 `data` 事件全量写盘；改为 `maybePersistProgress()` 每 **250ms** 落盘一次，循环结束补一次精确写入。

4. **概览合并策略分层**
   - 引擎层 `mergeDashboardTools(installed, detected)`：纯函数，已纳管优先，按 `tool+version` 或同 `path` 去重。
   - UI 层（浏览器 bundle）**不得 import 引擎 node 依赖**，故 mock 内联合并逻辑，`categoryLabel` 映射在 Dashboard 内镜像。

5. **⚠️ preload 桥接方法名必须与 `DevEnvApi` 接口严格一致（本次踩坑固化约定）**
   `src-ui/api.ts` 用 `Proxy` + `prop in real` 判断是否走真实桥接。若 preload 暴露的方法名与接口方法名不一致（如 `dashboardTools` vs `getDashboardTools`），该接口会**永久静默降级到 mockApi**，真实数据被丢弃。所有桥接方法名必须与接口逐一对齐。

6. **不要保留根级 `download/` 等沙盒/临时目录**
   排查用的临时数据根用完即清，避免污染仓库（用户明确约定）。

### B. 概览页「已安装工具」为空（最终根因）

- **现象**：概览页"已安装工具"显示"暂无"、KPI 为 0，系统检测工具不展示。
- **误判过程**（记录以避免再走弯路）：
  1. 怀疑 `loadDashboardTools` 无 try/catch 吞异常 → 加容错降级（无效）。
  2. 怀疑 handler `if (!cfg) return []` 早期 return 跳过扫描 → 改为始终 scanSystem（仍空，但这是正确的加固）。
  3. **真正根因**：`src-ui/api.ts` 的 `api` 代理用 `prop in real` 判断（`prop='getDashboardTools'`），而 `preload.ts` 暴露的方法名是 `dashboardTools` —— 名字不匹配导致该接口**永久降级到 mockApi**（mock 无配置时返回空）。这是 19 个桥接方法里**唯一名字不一致**的，故仅此卡片坏、其余正常。
- **修复**：`preload.ts` 方法名 `dashboardTools` → `getDashboardTools`（与接口对齐，IPC 通道名 `dashboard:tools` 不变）。
- **验证（终端直接跑，未依赖用户重启看截图）**：
  - 重建后 `grep -c getDashboardTools out/preload.cjs` = 2（新名已进产物）。
  - 纯 Node 复刻 handler 体（读 `devenv-root.txt` → `getCfg` → `scanSystem` → `mergeDashboardTools`）：本机扫出 **6 个系统工具**（JDK / Maven / Node.js 22.22.2 / MySQL / Redis 7.4.2 / Git 2.54.0），合并后 6 项全部 `source='detected'`。
  - `tsc --noEmit` 0 错误；`vitest` 47/47 通过。
  - 清理了排查期的调试日志与临时验证文件。
